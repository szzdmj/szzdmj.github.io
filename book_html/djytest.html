主图代理隔离测试

部署到能被 PHProxyPro 代理的源站,先直接打开确认 6 个变体全部显示,再通过代理打开, 记录哪几个变体的图消失/变占位。每个变体只隔离一个变量,消失的那个就是坏点。 图片用的是真实源站资源 i.epochtimes.com/.../id14803048-GettyImages-2284098649.jpg, 代理会按它处理真实页面的同样规则改写这些 URL。

#1 完整 <picture>(与源站主图 1:1)

webp source(/_next/image w=1200) + jpeg source(直连) + img 兜底(/_next/image w=1920)。浏览器通常选 webp source。

#2 只 webp source(相对 /_next/image, w=1200)

只保留浏览器实际会显示的那个源。若此变体消失,说明代理改写这个 _next/image 相对 URL 出了问题。

#3 裸 <img src=/_next/image w=1200>(无 picture/source)

去掉 <picture> 和 <source type=webp>,只留一个 img。用来判断是不是 <source type="image/webp"> 声明为 webp、代理却返回 jpeg 的类型不匹配导致的。

#4 直连原图 jpeg(绝对 i.epochtimes.com,非 _next/image)

源站 jpeg source 的形式。代理会把它改写成 /h1/<blob>/assets/...jpg 直通路径。若只有这个消失,问题在直通图路径(DR/leak-fallback)那条线。

#5 绝对 www.epochtimes.com/_next/image(w=1920)

源站 <img> 兜底那个 URL 形式(绝对 host + 内层 url 参数)。测代理对"绝对 _next/image URL"的改写。

#6 裸 img,无任何 class/style(排除 CSS 塌陷)

与 #3 同一 URL,但去掉 class/style。若 #3 消失而 #6 正常 → 是代理后 CSS 把容器/图样式改坏了,不是图本身。

判读:#2/#3 消失 → _next/image 相对 URL 改写问题;#4 消失 → 直通图路径;#5 消失 → 绝对 _next/image 改写; #3 消失但 #6 正常 → CSS 塌陷;#1 消失但 #2/#3/#4/#5 都正常 → <picture>/<source> 组合或 source 被 PHP 替换成占位。
